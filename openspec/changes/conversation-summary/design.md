## Context

当前系统的消息发送流程：

```
用户发送消息 → 保存用户消息 → 获取所有历史消息 → 发送给 LLM → 保存 AI 回复
```

问题：历史消息列表随对话长度线性增长，导致 token 消耗持续增加。

本设计实现滑动窗口 + 摘要压缩策略，在保留对话语义连续性的同时控制 token 消耗。

## Goals / Non-Goals

**Goals:**
- 实现滑动窗口机制，限制每次请求的消息数量
- 实现摘要压缩，保留早期对话的语义
- 支持增量更新摘要，避免重复处理历史消息
- 存储摘要到数据库，支持复用
- 对用户透明，无需前端改动

**Non-Goals:**
- 不实现异步摘要生成（采用同步模式，简化实现）
- 不单独存储代码块（通过 Prompt 引导保留代码）
- 不支持用户修改/删除历史消息后的摘要重建

## Decisions

### 1. 滑动窗口参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 窗口大小 | 20 | 超过 20 条消息触发摘要 |
| 保留原始消息数 | 10 | 最近 10 条保持原样 |
| 摘要更新频率 | 5 | 摘要之后每新增 5 条消息，更新摘要 |

**理由**: 基于主流 Chat 工具的实践和成本/效果平衡。

### 2. 摘要更新策略：增量更新

```
初次摘要: Summary1 = 压缩(M1-M10)
二次摘要: Summary2 = 压缩(Summary1 + M11-M15)
三次摘要: Summary3 = 压缩(Summary2 + M16-M20)
```

**选择增量而非全量的理由**:
- Token 成本低（~500+500 vs ~2000+500）
- 延迟小（~1s vs ~2-3s）
- 早期细节丢失可接受（早期对话通常已不重要）

**替代方案**: 全量重新处理
- 优点：摘要质量始终一致
- 缺点：成本高、延迟大、需要保留所有原始消息

### 3. 同步更新模式

```
用户发消息 → 检测是否需要更新摘要 → 同步生成摘要（阻塞） → 返回结果
```

**选择同步而非异步的理由**:
- 实现简单可靠
- 避免数据一致性问题
- 摘要生成约 1-2 秒，用户可接受

### 4. 摘要模型选择

使用用户当前对话的模型（复用 `Conversation.ProviderModelID`）。

**替代方案**: 使用更便宜的模型（如 gpt-4o-mini）
- 优点：降低摘要成本
- 缺点：增加复杂度，需要配置
- 决定：暂不实现，保持简单

### 5. 分组加权压缩

将参与摘要的消息分为三组，通过 Prompt 引导：

```
早期消息（前 1/3）：轻度概括
中期消息（中 1/3）：保留要点
近期消息（后 1/3）：保留完整细节（代码、数据、结论）
```

### 6. 数据模型

```go
type ConversationSummary struct {
    ID             uint      `gorm:"primaryKey"`
    ConversationID uint      `gorm:"uniqueIndex;not null"`
    Summary        string    `gorm:"type:text;not null"`
    EndMessageID   uint      `gorm:"not null"` // 摘要覆盖到的最后一条消息 ID
    CreatedAt      time.Time
    UpdatedAt      time.Time
}
```

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|---------|
| 摘要丢失重要细节 | Prompt 强调保留代码、数据、关键结论 |
| 摘要生成失败 | 降级：使用完整历史消息 |
| 摘要与实际历史不一致 | 可接受，用户可查看完整历史 |
| 增加首次响应延迟（1-2s） | 可接受，仅在有摘要更新时发生 |
| 多次压缩后细节丢失 | 可接受，早期细节通常不重要 |

## Migration Plan

1. 运行数据库迁移，创建 `conversation_summaries` 表
2. 部署新代码
3. 现有对话不受影响（新消息触发时自动创建摘要）
4. 无需回滚数据（新表为空）
